# PRD 06.1 - Router Prompt Hardening And Repo-Scoped GitHub Ingestion

> **Companion to:** `docs/prd/06-synthetic-conversation-eval-and-chat-reliability.md`
> **Prerequisite:** PRDs 01 through 03 should be complete. PRD 04 is recommended but not required.
> **Sequencing note:** This PRD is safe to run before PRD 04 and PRD 05. If PRD 04 is not complete, keep active-job prompt context optional and do not block router improvements on job-lifecycle features. PRD 05 is not required for this work.
> **Repository guidance:** Use `CLAUDE.md` for repo-wide workflow rules, `agents/CLAUDE.md` for routing work, and `tui/CLAUDE.md` for any UI/service touchpoints. Treat `.github/` instruction files as thin tooling mirrors, not the authoritative policy surface.
> **Prerequisite reading:** Read `agents/chat.py`, `tui/services.py`, `ingestion/github.py`, `llm.py`, `config.py`, and `test_smoke_formal.py` before starting.
> **Baseline test:** `python -m pytest test_smoke_formal.py -q` - all tests must pass before and after.
> **Core problem:** The current router prompt in `agents/chat.py` is a flat tool list with a few broad rules. It does not inject runtime state, does not clearly distinguish between tool execution and clarification, and gives the model too much room to answer vaguely. At the same time, the product only exposes username-scoped GitHub ingestion through `tui/services.py::ingest_github(username)`, so requests like "ingest data from this GitHub repo" cannot be handled precisely even with a better prompt. This PRD hardens the router prompt and adds a repo-scoped GitHub ingestion tool so plain-English repo requests become a real capability instead of a prompt guess.

---

## Design Rules

1. Treat prompt engineering as a routing contract, not as tone polishing.
2. Do not ask the model to infer a capability the product does not expose. Add the repo-scoped tool first, then teach the prompt to use it.
3. Prefer explicit action formats over free-form assistant text. The router should choose one of a small number of actions.
4. Keep account-level GitHub ingestion and repo-level GitHub ingestion distinct. They solve different user intents.
5. Do not rely on the LLM for cases that are cheap to parse locally, such as `owner/repo` refs and GitHub URLs.

  - active profile exists or not
  - active profile name
  - saved GitHub username if present
  - whether there is an active job and its title/company/status if available

- [ ] Keep the prompt short and structured. Use labeled sections such as:
  - `Role`
  - `Current state`
  - `Allowed actions`
  - `Tool guide`
  - `Examples`
- [ ] Remove generic wording like "be helpful" when it competes with routing precision

- [ ] Change the router instructions so the model must choose exactly one of these response envelopes:

- [ ] If the model returns malformed output, fall back safely by treating it as plain assistant text
- [ ] Add one explicit rule: when a required argument is missing, use `CLARIFY:` instead of guessing or calling the wrong tool
- [ ] Add one explicit rule: do not respond with both a clarification and a tool call in the same turn

### 3 - Rewrite tool descriptions around intent, not just names
- [ ] For each tool exposed to the LLM in `agents/chat.py`, rewrite the prompt documentation to include:

- [ ] Add explicit prompt guidance for ambiguous messages like:

## Background Context

<details>
<summary>Why prompt engineering alone is not enough here</summary>

The current `agents/chat.py` prompt tells the model to emit `TOOL_CALL:` lines, but it does not inject current runtime state and it does not clearly encode when to clarify versus when to execute. That makes the model more likely to over-answer or pick a nearby tool.

More importantly, the only GitHub ingestion tool exposed today is `run_ingest_github(username)`, backed by `tui/services.py::ingest_github(username)`. That capability fetches all repos for a user. It does not represent a single-repo ingest operation. So if the user says "ingest this repo", the router currently has no precise tool to call. Better prompt wording can reduce confusion, but it cannot create a missing product capability.

This PRD fixes both sides of the problem: it tightens the routing contract and adds the missing repo-scoped ingestion surface so plain-English repo requests can be handled deterministically.
</details>
